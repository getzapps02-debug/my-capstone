# Decision Log

## 2026-06-15 - Initial Product Direction

- Build a personal, local-first money-management application.
- Solve the problem of money falling short unpredictably.
- Make discovering hidden contributors to past shortfalls the primary user value.
- Use Neo4j to model connected financial evidence.
- Use line charts to show balances and spending trends over time.
- Make Guided Investigation the primary experience, supported by a compact dashboard.
- Use layered explanations: plain-language summary, expandable evidence, and synchronized graph/chart views.
- Include transaction replay as the MVP's distinctive visual interaction.
- Keep financial data local with Neo4j running in Docker.
- Support sample data, manual entry, and CSV import.
- MVP outcomes:
  1. Import CSV statements without duplicates.
  2. Show historical balance and category trends.
  3. Explain contributors to a past shortfall using connected evidence.
- Avoid unsupported causal claims; distinguish observed facts from inferred contribution.
- Defer future-shortfall prediction and rescue-plan comparison until after the historical model is reliable.

- None.

## 2026-06-15 - Party Mode Roundtable

- Confirmed a single-user local web application running on localhost.
- Confirmed Docker Compose as the packaging and startup approach.
- Clarified that the browser is the interface, not a cloud deployment model.
- Refined the promise from explaining "why" to investigating observable changes and contributors.
- Defined the initial user as a financially responsible adult with irregular expenses who cannot reconstruct a past shortfall.
- Required Neo4j to demonstrate inspectable evidence paths rather than serve as decorative visualization.
- Set measurable success criteria for duplicate prevention, fixture reconciliation, evidence traceability, explanation language, time to first investigation, local-only processing, and setup.
- Positioned the public artifact as a local-first financial explainability case study using synthetic data.
- Accepted Fast Path drafting with explicit assumption tags.

## 2026-06-15 - Assumption Review Approved

- Approved formative testing with five representative users.
- Expanded usability success beyond answer-finding: users must cite evidence, distinguish facts from inference, and recognize uncertainty.
- Established that any severe misleading interpretation forces revision and retesting regardless of aggregate results.
- Approved a public repository plus a recorded walkthrough of five minutes or less as the primary portfolio presentation.
- Defined the expected portfolio documentation set, synthetic sample data, clean setup, privacy review, and explicit limitations.
- Deferred public hosting unless hands-on access becomes necessary for evaluation.
